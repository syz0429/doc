# 基于 CARLA 的驾驶违规/交通标志检测数据采集项目说明

## 1. 项目概述

这个项目的核心代码位于 [main.py](D:/software/workspace/Git/nn/src/driving_violation_detection/main.py)，本质上是一个基于 **CARLA 自动驾驶仿真平台** 的数据采集脚本。  
它在仿真环境中完成以下工作：

1. 连接 CARLA 服务端并加载地图。
2. 生成背景交通流和主车辆。
3. 在主车辆上挂载 RGB 相机。
4. 从场景中的交通标志提取 3D 边界框，并投影到图像平面。
5. 对候选框进行筛选和非极大值抑制。
6. 保存图像以及对应的 XML 标注文件。
7. 自动切换天气，采集更丰富的训练数据。

从用途上看，这段代码更像一个 **交通标志检测数据自动采集器**。  
虽然目录名是 `driving_violation_detection`，但当前脚本的直接目标并不是判定“违规事件”本身，而是为后续的交通标志识别、场景理解、驾驶行为分析等任务生成数据集。

---

## 2. 项目背景与意义

在智能驾驶和计算机视觉任务中，交通标志检测是一个非常基础但又非常关键的能力。真实道路数据采集通常存在以下问题：

- 成本高，需要车辆、设备和人工标注。
- 天气、时间、道路条件难以系统控制。
- 某些长尾场景难以稳定复现。
- 标注耗时，尤其是检测框、类别和环境信息整理成本很高。

CARLA 这类高保真仿真平台的价值就在于：

- 可以快速构建可控的道路交通场景。
- 可以重复运行同一类测试。
- 可以通过程序自动生成标注。
- 可以方便地切换天气、地图、车辆和传感器配置。

因此，这个项目适合作为以下工作的前置环节：

- 交通标志检测数据集构建。
- 自动驾驶感知算法的预训练或验证。
- 多天气、多光照条件下的鲁棒性研究。
- 驾驶违规识别系统的环境感知基础模块开发。

## 3. CARLA 基础知识讲解

## 3.1 CARLA 是什么

CARLA 是一个面向自动驾驶研究的开源仿真平台，支持：

- 城市场景与道路地图仿真。
- 车辆、行人、交通灯、交通标志等交通要素建模。
- 摄像头、激光雷达、IMU、GNSS 等传感器仿真。
- Python API 控制仿真流程。
- 自动驾驶代理、交通管理器、场景复现和数据采集。

在这个项目中，CARLA 被用来做两件关键事情：

1. 构造可控的驾驶场景。
2. 自动生成交通标志图像及标注信息。

## 3.2 CARLA 的几个核心概念

### 1）Client / Server

CARLA 采用客户端-服务端架构：

- `carla.Client('localhost', 2000)` 用于连接本地仿真服务端。
- Python 脚本扮演客户端角色，负责发指令、取数据。
- CARLA 模拟器程序负责真正运行世界和物理仿真。

代码位置：`main.py:39-41`

### 2）World

`World` 可以理解为整个仿真世界，包含：

- 当前地图。
- 天气。
- 场景中的所有 Actor。
- 仿真设置。

常见操作包括：

- `client.load_world('Town05')` 加载地图。
- `world.get_settings()` 获取仿真设置。
- `world.apply_settings(settings)` 应用同步模式等参数。

代码位置：`main.py:94`, `main.py:98-101`

### 3）Map 与 Spawn Points

地图定义道路结构、路口、车道、交通灯和交通标志位置。  
`spawn_points` 是地图中预定义的可出生位姿，常用于刷车或放置主车。

代码通过：

- `world.get_map().get_spawn_points()`

获得车辆候选出生点。

代码位置：`main.py:110-111`

### 4）Actor

CARLA 中的车辆、传感器、行人等对象都属于 `Actor`。

本项目里主要包含三类 Actor：

- 背景交通车辆。
- 主车辆。
- 挂载在主车辆上的 RGB 相机。

代码位置：

- 背景车辆生成：`main.py:48-62`
- 主车辆生成：`main.py:126-134`
- 相机生成：`main.py:149-160`

### 5）Blueprint Library

Blueprint 可以理解为“对象模板库”。  
例如：

- `vehicle.*` 表示各种车辆模板。
- `sensor.camera.rgb` 表示 RGB 相机模板。

项目中先从 Blueprint Library 里选模板，再实例化 Actor。

代码位置：`main.py:49`, `main.py:124`, `main.py:127`, `main.py:150`

### 6）Traffic Manager

Traffic Manager 用于控制车辆自动驾驶行为，例如：

- 自动驾驶开关。
- 跟车距离。
- 车速偏移。
- 车道变更。
- 是否遵守信号灯。

在本项目中，背景车辆和主车辆都启用了自动驾驶，并对主车辆进行了额外行为设置。

代码位置：`main.py:103-105`, `main.py:117-118`, `main.py:137-148`

### 7）Sensor

传感器是 CARLA 的重要数据来源。  
本项目使用的是 `sensor.camera.rgb`，即 RGB 摄像头。

典型流程是：

1. 创建相机蓝图并设置分辨率、视场角。
2. 将相机挂载到车辆上。
3. 使用 `listen()` 注册回调函数。
4. 在回调里接收图像帧。

代码位置：`main.py:149-171`

### 8）同步模式

CARLA 可以工作在异步模式或同步模式下。

- 异步模式：模拟器自己往前跑，脚本被动接收数据。
- 同步模式：每次由脚本调用 `world.tick()` 推进一帧。

同步模式更适合数据采集，因为：

- 帧与帧之间更稳定。
- 传感器数据和场景状态更容易对齐。
- 便于做确定性实验和可重复采集。

本项目中：

- `settings.synchronous_mode = True`
- `settings.fixed_delta_seconds = 0.05`

表示每个仿真步长固定为 0.05 秒。

代码位置：`main.py:97-101`

## 5. 核心模块介绍

虽然项目没有拆分成多个 Python 文件，但从功能职责上可以划分为以下几个核心模块。

## 5.1 环境初始化模块

这一部分负责完成 CARLA 运行环境接入，包括：

这一模块的意义在于为后续所有 Actor、传感器和数据采集操作提供统一的仿真上下文。

## 5.2 场景交通流生成模块

函数 `spawn_vehicles(num_vehicles, world, spawn_points)` 用于生成背景车辆。

其逻辑是：

1. 从 `vehicle.*` 蓝图库中随机选择车型。
2. 从出生点中随机选择位置。
3. 调用 `world.try_spawn_actor()` 尝试生成。
4. 成功后加入列表，失败则打印提示。

之后，脚本把这些背景车统一交给 Traffic Manager 托管自动驾驶：

```python
for v in vehicles:
    v.set_autopilot(True, traffic_manager.get_port())
```

代码位置：

- 车辆生成函数：`main.py:48-62`
- 背景车辆实例化：`main.py:113-118`

这部分的作用是给主车辆提供一个更真实的交通环境，避免采集出来的图像过于理想化。

## 5.3 主车辆与自动驾驶控制模块

主车辆使用固定蓝图 `vehicle.audi.a2` 创建，然后启用自动驾驶。  
虽然代码中注释写着 “Disable Autopilot for manual control”，但实际执行的是：

```python
vehicle.set_autopilot(True, traffic_manager.get_port())
```

说明当前运行模式仍然是 **自动驾驶采集**，而不是人工驾驶采集。

此外，代码还配置了若干交通行为参数：

- `auto_lane_change(vehicle, True)`：允许自动变道。
- `set_global_distance_to_leading_vehicle(2.5)`：设置全局跟车距离。
- `ignore_lights_percentage(vehicle, 0.0)`：不忽略红绿灯，表示 0% 的概率闯灯。
- `vehicle_percentage_speed_difference(vehicle, -50)`：相对默认速度提高。
- `distance_to_leading_vehicle(vehicle, 3.0)`：为该车单独设置跟车距离。

代码位置：`main.py:126-148`

这一模块控制了主车在道路中的行为模式，也间接影响能否稳定遇到交通标志并完成采集。

## 5.4 传感器采集模块

项目挂载的是一个前向 RGB 相机，关键参数如下：

- 分辨率：`1024 x 1024`
- 视场角：`70`
- 安装位置：车辆前上方，`x=1, z=2`
- 俯仰角：`pitch=-3`

代码位置：`main.py:149-160`

相机数据通过回调函数异步写入队列：

```python
def image_callback(image):
    if not image_queue.full():
        image_queue.put(image)
```

随后主循环中再通过 `image_queue.get()` 取出最新图像进行处理。

代码位置：`main.py:162-171`, `main.py:503-520`

这种“回调 + 队列”的结构很常见，它把 **传感器接收** 和 **主线程处理** 解耦，减少阻塞。

## 5.5 交通标志检测与 3D 到 2D 投影模块

这是整个项目最核心的部分之一。

### 1）相机内参矩阵构建

函数 `build_projection_matrix(w, h, fov)` 根据图像宽高和视场角构建相机投影矩阵 `K`。

代码位置：`main.py:205-217`

### 2）世界坐标到图像坐标映射

函数 `get_image_point(loc, K, w2c)` 完成 3D 点到 2D 像素坐标的映射：

- 先把世界坐标点变换到相机坐标系。
- 再用相机内参矩阵投影到图像平面。

代码位置：`main.py:219-226`

### 3）交通标志框提取

函数 `get_signs_bounding_boxes(...)` 的主要流程是：

1. 遍历场景中的交通标志边界框 `TrafficSigns`。
2. 根据主车辆位置过滤掉过远目标。
3. 利用右向量点积，只保留车辆右侧目标。
4. 利用相机朝向点积，只保留相机前方目标。
5. 获取标志 3D 顶点并投影到图像。
6. 计算 2D 外接框。
7. 过滤越界框、小面积框、异常长宽比框。
8. 利用位置集合和冷却时间避免重复采集同一个标志。

代码位置：`main.py:247-295`

这里体现出几个很实用的工程思想：

- **几何筛选**：先做距离与方向过滤，降低无效框。
- **投影生成标注**：不用人工标注，直接由仿真几何信息生成。
- **重复样本控制**：通过 `captured_sign_locations` 和 `capture_cooldown` 提高数据多样性。

## 5.6 标注生成模块

## 5.7 天气调度与环境标签模块

## 5.8 后处理模块：IoU 与 NMS

交通标志在投影时可能生成多个重叠框，因此脚本实现了两个经典检测后处理函数：

### 1）IoU 计算

`compute_iou(box1, box2)` 用来计算两个框的交并比。

代码位置：`main.py:452-469`

### 2）非极大值抑制

`non_maximum_suppression(bboxes, iou_threshold=0.2)` 按面积从大到小排序，保留高质量框，去掉高度重叠的候选框。

代码位置：`main.py:471-486`

虽然这里不是深度学习模型输出的候选框，但在几何投影结果中同样可能存在重复框，所以 NMS 仍然有实际意义。

## 5.9 主循环与资源清理模块

主循环是整个程序的调度中心，主要负责：

- 调用 `world.tick()` 推进一步。
- 从队列中取图像。
- 根据时间切换天气。
- 计算交通标志框。
- 应用 NMS。
- 保存图像和 XML。
- 用 OpenCV 实时显示结果。
- 监听 `x` 键退出。

代码位置：`main.py:494-568`

程序结束后，`finally` 块会释放资源：

- 销毁 OpenCV 窗口。

代码位置：`main.py:570-577`

这部分非常重要，因为 CARLA 中如果 Actor 不清理，容易造成仿真环境残留对象，影响后续运行。
